Executive Summary

The Data Source Health Monitor Agent is a foundational operational agent that continuously oversees every data source connection configured within D365 Customer Insights — Data. In any CDP implementation, data freshness and reliability are non-negotiable; a single stale or broken connection can cascade into inaccurate unified profiles, degraded predictions, and flawed campaign targeting. This agent acts as an always-on sentinel, monitoring Dataverse, Azure Data Lake Storage, Azure Synapse Analytics, Power Query online connections, and third-party API-based sources.

By proactively detecting authentication failures, network timeouts, schema drift, and abnormal latency patterns before they manifest as downstream data quality issues, the agent dramatically reduces mean-time-to-detection (MTTD) from hours or days to minutes. It provides IT administrators and data engineers with contextual diagnostics — not just "connection failed," but precisely why, with recommended remediation steps. The agent also maintains a historical health ledger, enabling trend analysis and capacity planning for data infrastructure.

For organizations managing 10-50+ data source connections — common in enterprise CDP deployments — manual monitoring is impractical. This agent transforms reactive break-fix operations into a proactive, self-healing data pipeline posture, ensuring that the unified customer profile is always built on a foundation of current, complete, and trustworthy data.

Business Case & ROI

Problem Statement: Enterprise CDP deployments connect 15-40 data sources on average. Connection failures, expired OAuth tokens, schema drift, and network issues go undetected for hours or days, producing stale unified profiles and eroding trust in customer analytics.

Current State: Data engineers manually check connection logs each morning, often discovering failures only when downstream consumers report stale data. Average detection time for a data source failure is 6-12 hours. Each incident requires 2-4 hours of investigation and remediation, with cascading impacts on unification, segments, and exports.

Proposed State: The agent monitors all connections every 15 minutes, detects issues within minutes, provides root-cause diagnostics, and auto-remediates common failures (e.g., token refresh, retry with backoff). Escalation paths are pre-configured for issues requiring human intervention.

Quantified Benefits:

  • Data freshness improvement: 99.5% SLA compliance vs. current ~92%

ROI Timeline: Positive ROI within 8 weeks; full annualized ROI of 280-350% by month 6.

Executive Summary

The Schema Change Detection Agent addresses one of the most insidious risks in enterprise CDP operations: silent upstream schema changes that break data ingestion, corrupt unification logic, and degrade downstream analytics without immediate visible symptoms. In organizations where source systems are managed by independent teams — ERP, e-commerce, marketing automation, loyalty platforms — schema changes (new columns, removed fields, data type modifications, renamed attributes) occur frequently and often without advance notice to the CDP team.

This agent establishes a proactive defense by continuously comparing source system schemas against the last-known-good schema snapshot stored in Customer Insights. When drift is detected, the agent classifies the change type (additive, breaking, or semantic), assesses its potential impact on existing column mappings, match rules, and merge policies, and generates a recommended response plan. For non-breaking additive changes, it can auto-suggest updated mappings. For breaking changes, it pauses the affected ingestion pipeline and alerts the data engineering team with full context.

The agent integrates with Azure DevOps or GitHub to create pull-request-style change reviews for schema modifications, enabling data teams to adopt a "schema-as-code" discipline. Over time, it builds an institutional knowledge base of schema evolution patterns per source, enabling predictive alerts when a source system is likely to introduce changes based on historical release cadences.

Business Case & ROI

Problem Statement: Upstream schema changes in source systems (ERP, e-commerce, CRM, loyalty) frequently break CI data pipelines. Without proactive detection, broken ingestion silently degrades unified profiles, segments, and predictions — often discovered only when business users report anomalies.

Current State: Schema changes are discovered reactively when a data refresh fails or when analysts notice data gaps. Average time to detect a breaking schema change: 1-3 days. Each incident requires 4-8 hours to diagnose, re-map columns, test, and re-run unification. Organizations experience 3-6 such incidents per quarter.

Proposed State: The agent pre-validates schemas before every refresh, detects drift instantly, classifies impact severity, and provides auto-generated remediation plans. Breaking changes halt ingestion proactively, preventing downstream corruption.

Quantified Benefits:

  • Prevention of corrupted unification runs: $150,000-300,000/year in avoided downstream rework
  • Reduced incident investigation time: 4-8 hours to 30 minutes per incident (90% reduction)
  • Eliminated silent data degradation: Prevent 10-20 days/year of operating on stale/corrupt profiles
  • Data engineering productivity: 15-20% time savings for data engineering team
  • Improved trust in CDP data: Measurable increase in business user adoption of CI-driven insights

ROI Timeline: Positive ROI by week 10; annualized ROI of 250-320% by month 8.

  3. As an IT administrator, I want schema change history tracked in a version-controlled ledger, so that I can audit what changed, when, and who approved the mapping update.

Agent Type: Pre-emptive validation agent with schema versioning and change management workflows

Processing Logic:

  2. Compare current schema against the last-known-good snapshot using structural diff algorithm
  3. Classify each change: Additive (new column), Destructive (removed/renamed column), Type Change (data type modification), Semantic (same name but different meaning based on value distribution)
  4. For each detected change, trace downstream dependencies: affected column mappings, match rules, merge policies, measures, and segments
  5. Score impact severity based on number and criticality of affected downstream components
  6. For non-breaking additive changes: auto-generate mapping suggestions and queue for review
  7. For breaking changes: pause affected data source refresh, generate detailed impact report, create remediation work item
  8. Update schema snapshot repository with new version and change metadata
  9. Notify affected stakeholders based on their role and the components they own

Output Actions:

  • Auto-generated column mapping suggestions for additive changes
  • Data source refresh pause/resume controls
  • Azure DevOps or GitHub work items for schema remediation
  • Teams notifications to affected data engineers and business analysts

Feedback Loop: Track schema change frequency per source, accuracy of impact severity classification, adoption rate of auto-suggested mappings, and time-to-remediation. Refine classification models and build predictive alerts based on source system release cadences.

Executive Summary

The Match Rule Effectiveness Agent is a quality-assurance agent focused on the most critical step in any CDP: identity resolution through matching. D365 Customer Insights — Data relies on configurable match rules — exact matches on email, fuzzy matches on name + address, probabilistic matches on phone + postal code — to determine which source records represent the same real-world customer. The quality of these rules directly determines the accuracy of every unified profile, segment, prediction, and activation downstream.

Yet most organizations configure match rules once during initial deployment and rarely revisit them. Over time, data patterns shift: new source systems introduce different identifier formats, data quality degrades in specific sources, customer behavior changes create new identity fragmentation patterns. Without continuous evaluation, match rules that were once optimal become either too aggressive (creating false merges that combine distinct customers) or too conservative (leaving the same customer fragmented across multiple profiles).

This agent systematically evaluates each match rule's precision and recall by sampling unified profiles, analyzing cluster characteristics, and cross-referencing against known-truth datasets. It identifies rules producing excessive false positives, rules with diminishing returns, and gap areas where no current rule captures obvious matches. The agent delivers actionable recommendations: tighten this threshold, add a blocking key, retire an underperforming rule, or introduce a new rule targeting a specific fragmentation pattern.

Business Case & ROI

Problem Statement: Match rules configured during initial CDP setup degrade over time as data patterns evolve. Over-matching creates merged profiles combining distinct customers (causing privacy violations and confusing experiences), while under-matching leaves customers fragmented (reducing CLV accuracy and campaign effectiveness).

Current State: Match rule quality is reviewed manually once or twice per year, typically triggered by business user complaints. No systematic measurement of precision/recall exists. Organizations typically operate with 5-15% over-matching and 10-20% under-matching rates without awareness.

Proposed State: The agent continuously evaluates match rule effectiveness using statistical sampling, cluster analysis, and known-truth benchmarks. It provides per-rule precision/recall metrics, identifies degradation trends, and recommends specific tuning actions with projected impact.

Quantified Benefits:

  • Improved unification accuracy: 5-15% reduction in false matches, 10-20% reduction in missed matches
  • Revenue impact: $500,000-2,000,000/year from improved CLV accuracy and campaign targeting (varies by org size)
  • Privacy risk reduction: 60-80% fewer incorrect profile merges that could expose data to wrong customers
  • Operational efficiency: 70% reduction in manual profile correction requests from business users
  • Prediction model improvement: 8-12% lift in churn and CLV prediction accuracy from cleaner profiles

ROI Timeline: Measurable improvement by week 12; full ROI of 400-600% by month 12.

User Stories

  1. As a data engineer, I want per-rule precision and recall metrics after every unification run, so that I can identify which rules need tuning and prioritize my optimization efforts.
  2. As a CRM manager, I want alerts when match quality degrades beyond acceptable thresholds, so that I can pause dependent campaigns until identity resolution is corrected.
  3. As a data privacy officer, I want reports identifying potential false merges where distinct customers may have been combined, so that I can ensure compliance with data protection regulations.
  4. As a marketing analyst, I want visibility into which customer segments are most affected by matching quality issues, so that I can adjust targeting strategies accordingly.
  5. As a data engineer, I want the agent to recommend specific rule adjustments (threshold changes, new blocking keys, rule additions), so that I have concrete actions rather than abstract quality scores.
  6. As a CMO, I want a monthly identity resolution quality scorecard, so that I can track the reliability of our customer count and segment sizes over time.
  7. As an IT administrator, I want the agent to simulate proposed rule changes against historical data before deployment, so that we can validate improvements without risking production data quality.

Agent Architecture & Design

Agent Type: Analytical evaluation agent with simulation capabilities and recommendation engine

Input Sources:

  • D365 Customer Insights match rule configurations and parameters
  • Unified profile clusters (output of the matching process)
  • Source record data participating in unification
  • Historical unification run metadata (cluster sizes, merge ratios)
  • Optional: known-truth datasets or manually verified match/non-match pairs

Processing Logic:

  1. After each unification run, extract a stratified sample of unified profile clusters (small, medium, large, suspicious)
  2. For each sampled cluster, evaluate evidence strength: how many match rules contributed, confidence scores, attribute agreement rates
  3. Calculate per-rule metrics: match contribution rate, exclusive match rate (profiles only matched by this rule), false-positive indicators
  4. Identify suspicious clusters: unusually large clusters, clusters spanning unlikely demographics, clusters with contradictory attributes
  5. Compute overall precision estimate using cluster quality heuristics and known-truth benchmarks where available
  6. Detect fragmentation by identifying source record pairs with high similarity but no match rule firing
  7. Generate rule-level recommendations: tighten/loosen thresholds, add/remove blocking keys, retire underperforming rules, propose new rules
  8. Simulate recommended changes against historical data to project impact on cluster count, average cluster size, and known-truth metrics
  9. Produce effectiveness report with visualizations, metrics, and prioritized action items

Output Actions:

  • Per-rule effectiveness scorecards with precision/recall estimates
  • Suspicious cluster reports for manual review
  • Ranked recommendations with simulated impact projections
  • Trend reports showing match quality evolution over time
  • Integration with CI match rule configuration for one-click threshold adjustments

Feedback Loop: Track which recommendations are adopted, measure actual vs. projected impact of changes, and refine the recommendation engine based on outcomes. Incorporate manual match/non-match verdicts from cluster reviews as additional training signal.

Executive Summary

The Cross-Source Identity Resolution Agent tackles the CDP's most fundamental challenge: stitching together a single, accurate view of each customer from fragmented records scattered across dozens of disconnected systems. While D365 Customer Insights provides native unification capabilities, the complexity of real-world identity resolution — where the same customer might appear as "john.smith@gmail.com" in the email system, "John A. Smith" in the CRM, loyalty card #4489221 in the POS system, and a cookie-based anonymous visitor in the web analytics platform — demands continuous, intelligent oversight that goes beyond static rule configuration.

This agent monitors the identity graph produced by unification, identifies fragmentation patterns (the same customer split across multiple profiles), and detects emerging identity signals that could improve resolution. It goes beyond the Match Rule Effectiveness Agent by focusing specifically on the identity graph topology: analyzing connected components, identifying weak links, discovering bridge identifiers, and recommending graph-based resolution strategies that complement traditional rule-based matching.

The agent is particularly valuable in omnichannel environments where customers interact through many channels, often without providing consistent identifiers. By applying probabilistic matching, graph traversal, and behavioral similarity analysis, it can suggest matches that no single deterministic rule would catch — for example, linking an anonymous web session to a known customer based on behavioral fingerprinting and device graphs. This dramatically improves the completeness of unified profiles, enabling more accurate CLV calculations, better churn predictions, and more personalized customer experiences.

Agent Architecture & Design

Agent Type: Continuous analysis agent with identity graph reasoning and supervised merge recommendations

Input Sources:

  • Unified profile identity graph (cluster membership and linking evidence)
  • All source records participating in unification with their identifiers
  • Activity/interaction data (behavioral signals for probabilistic matching)
  • Customer Insights match rule configuration and match evidence logs
  • Optional: device graphs, cookie-to-customer mappings, third-party identity resolution signals

Processing Logic:

  1. Construct the full identity graph from unification output — nodes are source records, edges are match evidence
  2. Analyze graph topology: identify weakly connected components, isolated nodes, and bridge identifiers
  3. For isolated/orphan profiles, search for potential matches using expanded similarity criteria (phonetic name matching, address normalization, behavioral fingerprinting)
  4. Apply graph-based resolution: if Record A matches Record B, and Record B matches Record C, evaluate transitive merge of A-B-C with confidence decay
  5. Detect bridge identifiers: fields that uniquely connect records across sources that no current match rule exploits
  6. Score each potential merge recommendation: combine deterministic evidence, probabilistic similarity, behavioral correlation, and graph centrality
  7. Classify recommendations: High confidence (auto-merge eligible), Medium (recommend with review), Low (flag for investigation)
  8. Generate merge recommendation queue with evidence bundles for human reviewers
  9. Track resolution trends: fragmentation rate over time, resolution by source pair, bridge identifier effectiveness

Output Actions:

  • Scored merge recommendations in a review queue with evidence bundles
  • Auto-merge execution for high-confidence recommendations (with undo capability)
  • Bridge identifier discovery reports with match rule suggestions
  • Fragmentation trend dashboards in Power BI
  • Identity graph visualization for investigative analysis

Feedback Loop: Capture human accept/reject decisions on merge recommendations to improve scoring models. Track post-merge data quality metrics to detect problematic merges. Use reviewer feedback to refine confidence thresholds and discover new resolution patterns.
